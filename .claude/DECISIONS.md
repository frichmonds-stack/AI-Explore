## Stack
- **Vite over CRA** — CRA is deprecated and incompatible with Tailwind v4's PostCSS pipeline (`@plugin` directives pass through unprocessed). Vite handles Tailwind v4 correctly out of the box.
- **DaisyUI over shadcn** — DaisyUI uses CSS classes rather than installed React components, making it lighter on tokens and better suited to a content-heavy static site.
- **Static JSON content** — all content lives in `/frontend/src/content/*.json`. No CMS or backend needed at this stage. Easy to edit without touching components.
- **React Router v7 client-side routing** — single-page app feel with no server required. Fits static hosting.

## Strategy (2026-07-20 — the core premise: four questions and a stance)

*Recovered 2026-07-21 from the 2026-07-20 "Infrastructure building" session transcript — the design work happened in discussion turns hours before any code, and the session's Publish Close recorded only the diff. See the Process note at the foot of this section.*

- **The premise is the site's spine, not just the benchmark's preamble.** Owner reframe: it must be "simple enough anyone can fit in, deep enough to actually improve performance." Named trap: you cannot maximise both *in the same layer* — go broad and you get the AITSL standards (universal, used by nobody to improve); go deep and only true believers adopt it. Resolution: **a simple surface over a deep core** — a handful of plain questions any teacher already half-asks, each opening into sourced principles. Same progressive-disclosure logic as the tools UI.
- **The model: four questions and a stance.** Every facet of the work — unit plan, worksheet, marking pass — is the same four questions at different grain sizes:
  1. *What exactly am I trying to teach?* — curriculum fidelity, learning intentions, accuracy
  2. *How will I get it into their long-term memory?* — CLT, Rosenshine, Willingham, explicit instruction
  3. *How will I know, and what will I do about it?* — formative assessment, checking for understanding, Wiliam
  4. *Who isn't with me, and what needs to change?* — differentiation, responsive teaching, UDL

  Wrapped in a **professional stance**: the teacher holds the judgement; theories and tools (including AI) are instruments, not authorities. Improvement = asking the questions more precisely and answering them with better evidence.
- **Research areas become tools for jobs, not competing camps** — CLT isn't an ideology you join, it's the best available answer to question 2. Mirrors the site's existing editorial philosophy for AI tools.
- **The improvement model and the benchmark rubric are the same object viewed from two sides** — the six criteria map near one-to-one onto the four questions (curriculum fidelity → Q1, instructional design → Q2, assessment integrity → Q3, teacher voice → the stance). One spine; everything cites it. *(AI proposal, owner did not rule on it.)*
- **Scope — three rings, one test** (owner raised the question: do management, relationships, admin, parent contact, marking, feedback belong in the core?):
  1. **The core loop** — the four questions. Teaching itself.
  2. **The conditions** — classroom management and relationships. Not instruction, but load-bearing: the loop cannot run without them.
  3. **The work around the work** — admin, parent contact, compliance. Outside teaching, admitted by one test: **does it serve the loop?** Efficiency on admin is only worth something if the reclaimed hours flow back into the core.

  **Marking and feedback are not outside** — they *are* question 3 (marking gathers the evidence, feedback acts on it). A model excluding them wouldn't be a teaching model.
- **OWNER CALL — rings 2 and 3 are acknowledged and deferred, not theorised now.** Rationale: priority and opportunity cost. None of the nine benchmark tasks need a theory of classroom management or parent contact behind them; writing one now is effort spent on chapters nothing cites. **Rider:** the deferral must appear as a *declared scope statement inside the premise document*, not as silence — same trust mechanism as the bias declaration. Without it the first sceptical reader lands the easy punch ("you've reduced teaching to instructional design"); with it, it's a stated, arguable boundary rather than a blind spot.
- **OWNER CALL — surface the deferral as a "coming soon" feature, not a disclaimer strip.** Owner's reframe on an AI proposal for a caveat band: a disclaimer is apologetic, a visible roadmap item is a trust signal. Shipped as the fifth homepage pillar card ("The whole job" — behaviour management, relationships, parent contact, admin; muted, dashed, non-navigating, **"Coming soon"** wording chosen by the owner over "under construction"). Logged noted risk: coming-soon elements can read as "unfinished" on a young site; judged acceptable because the other four pillars are full.
- **Contestable — recorded in the 2026-07-13 entry below as settled, but never actually examined by the owner:** the specific six criteria, the nine tasks and their learning-area/band pinnings, the line "pedagogy is not a seventh criterion, it is the authority the criteria stand on," and the exact contents of the bias declaration. Treat all four as *proposed, awaiting ruling* — the step-1 draft must present them as such rather than citing them back as decisions.
- **Drafting discipline confirmed:** step 1 (premise) only, then owner red-pen, *then* steps 2–3 (rubric descriptors, prompt pack). Compounding risk: if the premise framing is rejected, anything built on it needs rework.
- **Process note (2026-07-21):** this entry existed only in a session transcript for a day, and was recovered by chance. The gap it exposes: `close-out.md` prompts for what changed *on disk*, so decisions reached in discussion turns — especially in a session that later gets consumed by an unrelated problem — can pass through a Publish Close unrecorded. See `close-out.md` step 2.

## Strategy (2026-07-13 — the benchmark: a stated theory of good teaching, operationalised)
- **Core premise document first** — a one-page position statement: what good teaching is (one-paragraph premise), 6–8 sourced universal principles, and an explicit **bias declaration** ("the position I come from": leans explicit/structured instruction for novices, weights cognitive-science evidence — Willingham, CLT, Rosenshine — over discovery traditions; written by a practising K–12 Australian educator; names what it is *not* claiming — inquiry/play/agency succeed *on top of* these foundations, not never). Manufactured neutrality rejected; a stated, sourced, arguable leaning is the trust model. Lives on its own page (`/approach` or About spine); methodology, rubric, and pedagogy track all cite it. The pedagogy track needs an audit pass to align with it once written.
- **Benchmark structure: 6 criteria × 9 tasks × applicability matrix.** Criteria = a unified "excellent practice" rubric (curriculum fidelity · instructional design · accuracy · assessment integrity · classroom readiness · teacher voice & judgement), each 0–4 with descriptors written in teacher-natural language, sourced to the premise's principles. **Pedagogy is not a seventh criterion — it is the authority the criteria stand on.** Differentiation is deliberately a *task*, not a criterion (owner call). Accessibility basics fold into instructional-design descriptors.
- **Tasks: 9, each pinned to one learning area + one band** — the 8 Australian Curriculum learning areas map one-per-task (lesson plan, unit plan, graphic, prose piece, worksheet, assessment, marking against an assignment sheet, PowerPoint) + a 9th, **differentiate an existing task** (works from supplied input — the most authentic real-world AI use). Bands staggered across primary/middle/senior; **ECE deferred as a named suite extension**. Tools graded only within their claimed scope (applicability matrix); general assistants face all nine.
- **Prompts embody best practice, in plain teacher voice — never framework names.** Owner call, reversing an interim "neutral prompts" position: prompts state what good looks like in plain language ("clear learning intention, model it first, check who's got it, keep the rigour") because teachers *should* prompt from that headspace and it eliminates the "AI didn't know" excuse — only "can't do it" remains. The prompt pack doubles as an exemplar resource teachers copy. Prompts are versioned; every run is date- + model-version-stamped (staleness accepted as "dated snapshot" — re-runs become trend content).
- **The plain-vs-intentful prompt gap is content, not infrastructure** — a one-off comparison article (2–3 tasks, plain vs best-practice prompts, outputs side by side) proves the site's thesis with receipts; the standing benchmark stays single-condition.
- **Benchmark results back the pedagogy badges** — tool-card claims like "Strong for CLT" graduate from authored placeholders to evidence-backed links pointing at the graded artefacts. Published outputs (the artefacts themselves, side by side) are the flagship shareable content; scores are the guide rail, not the claim.
- **Drafting order: (1) core premise + principles → (2) criteria descriptors → (3) prompt pack** — each derives from the last. Suite-v2 candidate ideas (EAL/D translation, IEP drafting, dual-band lesson-plan variant) get listed in the spec, not built.
- Also this session: owner intends a first-person **opinion piece** (the espoused-but-ignored-at-work convictions) — pairs with the premise document and the implementation theme.

## Strategy (2026-07-13 — launch gate, ratings sequencing, leadership wedge)
- **Launch gate: at least 5 solid tool reviews + 2 articles** (one of the two = a methodology piece, "How we review AI tools for Australian classrooms", which every review links back to as the credibility anchor). Launch when that bar is met, not before.
- **Ratings are sequenced, not built in tandem with accounts.** Crowd ratings at launch would show noise (cold start) and full accounts add backend + Australian privacy burden on a child-safety-branded site. Sequence: (1) editorial verdicts in tool reviews now → (2) anonymous "Was this useful? / My school uses this" pulse widget (Cloudflare Worker + KV, IP rate-limited, displayed as social proof not authority) at/after launch → (3) verified educator ratings via school-email magic link only when traffic proves demand. Positioning note: the site's trust model is editorial judgement, not aggregation — crowd scores stay secondary.
- **"AI implementation in schools" is the leadership wedge** — content for deputy principals / digital-learning leaders (staged adoption playbook: explore → pilot → evaluate → policy → PL; translation of the national Framework for Generative AI in Schools; pilot-evaluation template; governance/privacy checklist). Starts as a tagged theme inside Articles (no new nav pillar pre-launch); promote to a "For leaders" section only if it proves to be a traffic driver. This revives the deferred school-policy section.

## Strategy (2026-06-22 — work is the hook, pedagogy is the destination)
- **Pedagogy is the focus; work is the hook — not a change of focus, a change of entry.** (Clarified by the owner: "I didn't want to change the focus; it still should be pedagogy first BUT people come in looking for work first… the site should pull them towards greatness.") Read "work-first" as *first encountered*, not *most important*. The front door is "what do you need to do?"; the site's job is the **pull** from "get it done fast" to "teach it brilliantly". Every work surface must bend toward better teaching and never dead-end at efficiency. Full doc: `.claude/project-context.md`.
- **The test for any feature:** does it leave the educator more thoughtful about their practice than when they arrived? Efficiency is the bait; craft is the catch.
- **Guides are the hook that carries the pull** — short task walkthroughs (`guides.json`, `/guides`) that *start* at efficiency and *end* at craft: the `pedagogyNote` is deliberately the **closing elevation** (pine gradient, "the craft behind it"), so a guide ends on the teaching, not a checklist. Tools answer "what could I use?"; Guides answer "how do I do this well, safely — and become a better teacher doing it?". Every guide carries a mandatory child-safety block, copy-paste prompts, a verify checklist, and an "explore next" rail.
- **Homepage carries the pull** — hero ("start with the job… let it pull you towards better teaching"), a pine "why these guides" doorway band linking to the pedagogy track, and pedagogy elevated in the pillar grid with a "Where it leads" badge. Nav order is entry-ordered (Guides → Tools → Learn AI → Teach well → Use it safely → Explore); pedagogy's primacy is shown by gravity, not nav position.
- **Audience broadened to K–12 educators generally** (was CEWA-only), but **CEWA badges retained** as the worked example of approval/safety diligence.
- **Guides reuse the Tools taxonomy** — no duplicate filter vocabulary; `guides.json` only adds `difficulties`. Guides cross-link to real tool ids and inherit their CEWA badge.
- **Child safety is structural, not editorial** — it's a required field on the guide content model, surfaced on every guide and on the homepage, so it can't be omitted by authoring.

## Content
- **CEWA approval layer HIDDEN (2026-07-01) — do not re-add without authorisation.** The tool approval statuses came from an **internal (non-public) CEWA source** — three searches (Claude, Gemini, owner) found no public version. Publishing that data on a public personal site is a confidentiality + professional risk, so the whole CEWA/approval layer is switched off via `frontend/src/config.js` (`SHOW_APPROVAL_STATUS = false`): all badges, the approval filter, the status legend/footer, and CEWA-framed copy are hidden; every tool's `cewaStatus` reset to `not-reviewed`; `cewaProvided`/`accessNote` cleared; the raw `cewa-status.json` deleted; internal specifics scrubbed from tool `notes`/`descriptions`; all user-facing "CEWA"/"Catholic" wording removed. **Verified: `dist/` contains no "CEWA"/"Catholic"/internal tells.** Reversible: if an authorised/public source is confirmed, flip the flag to true and repopulate statuses (prior values recoverable from git history). Note: the internal values still exist in git *history* (private repo) — scrub history before any public repo. Supersedes the earlier "CEWA badges retained / full landscape with status" decisions below.
- **Website, not web app** — read-and-explore resource. No quizzes, no progress tracking, no accounts.
- **AI is neutral — intentionality is the point** — core editorial philosophy. AI isn't good or bad; unfettered access without pedagogical intent is the problem (same mistake as devices + social media). The resource models purposeful, precise use.
- **Tools are the hook, pedagogy is the ambition** — tools bring teachers in, but the resource aims to move them from "what tool should I use" to "how do I teach better." Every tool links back to pedagogical practice.
- **Pedagogy badges on tool cards** — each tool carries badges for the teaching frameworks it best supports (e.g. "Strong for: Cognitive Load Theory"). Badges link through to the pedagogy section. Makes tools filterable by pedagogical approach. Full star-rating system deferred to Phase 2.
- **Every tool gets its own detail page** — route `/tools/:toolId`. Each tag on the card (role bucket, pedagogy framework) gets its own section on the detail page explaining *why* this tool suits that role or pedagogical approach and *how* to use it that way. The card is a summary; the detail page is the substance.
- **Card has two click targets (2026-06-22)** — the tool **name is a direct link** to `/tools/:toolId` (underlined + diagonal arrow = visible "clickable" affordance); the rest of the card surface opens the quick preview. Implemented as a stretched `.lmn-tool__select` button behind the content (content `pointer-events:none`, title/tags `pointer-events:auto`) so the two targets don't nest. The old "In the spotlight" cue label was removed (now "Quick preview" / "Previewing"). Rationale: clicking a tool should be able to go *straight* to its page, not only into the preview.
- **Persistent spotlight, not in-grid expansion** — tried an in-grid FLIP morph (card expands in place to full width, grid reflows) but it felt janky. Replaced with a dedicated sticky spotlight panel above the grid. Cards are calm scannable *selectors* (logo, name, vendor, secondary status, two-line synopsis, three fixed tag slots: use → role → pedagogy); clicking one drives the spotlight. No grid reflow = no jank. The spotlight auto-selects the first featured tool on load, is sticky, expands on click, and minimises to a compact bar on scroll (re-expands on next click or bar tap). Progressive disclosure: detailed setup/risks/pricing stay on the tool detail page. Implementation notes: `prevent​Default` on card mousedown stops focus-scroll; minimise-on-scroll re-baselines after the expand animation so scroll-anchoring (which bumps scrollY to keep the grid stable when the panel grows) isn't misread as a user scroll. Respects `prefers-reduced-motion`.
- **Use-category is the primary tag axis** — teachers search by the job to be done ("Lesson Planning", "Assessment & Feedback"), so `useCategories` is its own data field and the strongest (filled pine) chip on the card, distinct from role and pedagogy. Added a "What you need" filter as the first filter row.
- **Domain → Work type hierarchy in filters (2026-06-25)** — the old "Work area" facet (the flat `roles` list) was renamed **Domain** and made the **parent** of **Work type** (`useCategories`): Domain now leads the filter bar and choosing one **narrows** the Work type options (and clears a now-invalid Work type). Mapping lives in `tools.json` `meta.domains` (6 domains → their child `useCategories`); a tool/guide belongs to a domain if any of its work types falls under it. Applied to both ToolsPage + GuidesPage (shared `FacetFilters`), the glossary (`domain` group, before work-type), and `definitions.js` (tooltips). Other label tidy-ups same day: pedagogy facet shows **full** framework names (e.g. "Universal Design for Learning", not "UDL"); Year level shows the band range and renamed bands ("Primary School (3–6)", "Senior School (10–12)"). **Known follow-up:** tool/guide **cards** and tool **detail pages** still show the legacy `role` chip/sections — `roles` data was left in place; migrating those surfaces to `domains` is a separate task (see BACKLOG).
- **Approval status: text + restrained colour, never colour alone** — visible labels are exactly `Approved / Conditional / Unreviewed / Not Approved` (plus `Under Review`), no "CEWA" in the badge text. Conditional = amber, kept visually secondary to tool name + synopsis.
- **"Strongest" tag is authored-order for now** — no relevance/ranking data exists yet, so the collapsed card shows the first array item per category as a placeholder. Real ranking is a planned data-model addition (see BACKLOG).
- **Tag / guide links point to placeholders** — `/explainer/:category` (general uses/roles/pedagogies explainers) and `/tools/:toolId` (full guide) are lightweight "in development" placeholder pages until the real content is built. Keeps interactions real and testable without inventing final content.
- **Risks first** — child development risks surfaced throughout, not buried at the end.
- **Teacher role as primary axis for tools** — buckets are Administration, Classroom, Curriculum, Assessment, Professional Learning, Creativity & Media, Prohibited. Teachers think in tasks, not AI capabilities.
- **Tools not limited to CEWA list** — full tool landscape shown with CEWA status badge on each. Teachers on personal devices can make their own call.
- **Backend deferred** — no server until content becomes dynamic or ratings/tracking needed.
- **School policy section deferred** — planned for a later phase.
